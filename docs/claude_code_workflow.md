# Claude Code 開発ワークフロー

このドキュメントは、Claude Code が今後の開発で守るべき共通ルールをまとめたものです。

---

## 1. 作業開始

作業を始める前に必ず以下を確認する。

- `git status` で変更状態を確認
- `git pull` でリモートの最新状態を取得
- 現在のブランチが正しいか確認
- 作業内容（ユーザーの指示）を再確認してから実装を始める

---

## 2. 開発中

- **小さな単位で実装する**：一度に大きく変えず、動作確認できる単位に分ける
- **既存動作を壊さない**：変更前に影響範囲を把握し、既存機能への副作用を確認する
- **必要ならE2Eテストを実施する**：実際に動かして確認できる場合は実行する
- **APIキーや .env の内容は絶対に表示しない**：ドメインや存在確認に留める
- **大きな変更の前は提案する**：仕様変更・構造変更はユーザーの了承を得てから実施する

---

## 3. 報告フォーマット

作業完了時は必ず以下の順番で報告する。

1. 作成・変更したファイル
2. 変更内容の概要
3. テスト結果
4. `git status`
5. コミットしたか
6. push 待ちか

---

## 4. 実測主義

**最重要ルール：実際に確認できていない内容を推測で報告しない。**

- 確認できない場合は「確認できませんでした」と正直に報告する
- バックグラウンド実行の出力が読み取れない場合は、ユーザーにその旨を伝え、確認方法を提案する
- もっともらしい内容であっても、実測していない値（URL・件数・ファイル名など）は報告しない

---

## 5. push 前チェック

push を実行する前に以下をすべて確認する。

- [ ] `git status` でワーキングツリーが clean であること
- [ ] ブランチが正しいこと（通常は `main`）
- [ ] `origin/main` との差分（何コミット先行しているか）を確認
- [ ] テストが成功していること
- [ ] `.env`・APIキー・パスワードがコミットに含まれていないこと

---

## 6. 作業終了

作業を終えたら以下を確認して終了レポートを出す。

- `git status` でクリーンであること
- 最新コミットID と内容を確認
- push 済みか・push 待ちかを明示
- ユーザーに終了を報告する

---

## 7. ChatGPT との役割分担

| 役割 | 担当 |
|---|---|
| 設計・アーキテクチャ | ChatGPT |
| 技術選定・レビュー | ChatGPT |
| 長期計画・方向性の決定 | ChatGPT |
| 実装・リファクタリング | Claude Code |
| テスト・動作確認 | Claude Code |
| Git 操作（commit / push） | Claude Code |
| ドキュメント更新 | Claude Code |

---

## 8. 提案ルール

- より良い設計や改善案を思いついた場合は積極的に提案する
- ただし、ユーザーの了承なしに仕様変更・大規模変更は行わない
- 提案は「提案です」と明示し、実装は了承を得てから行う

---

## 9. 品質方針

- **可読性を優先**：難しい書き方より、読んで分かるコードを書く
- **保守性を優先**：後から変更しやすい構成にする
- **小さく実装・小さくテスト**：動作確認できる単位で進める
- **ドキュメントも同時更新**：README・CHANGELOG・ROADMAP は実装と合わせて更新する
